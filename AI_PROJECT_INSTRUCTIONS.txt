Purpose
- Keep Plaid sync, categorization, budgets, goals, and financial UI behavior consistent.
- Preserve user corrections (categories, merchant rules, splits) and never expose bank tokens on device.
- Ship a reliable personal finance app with minimal taps and actionable errors.

Product North Star
- Part of the broader life-optimization vision (see `../Optimized/AI_PROJECT_INSTRUCTIONS.txt`); **Budget Tracker is the finance domain**.
- Primary objective: maximize usefulness while minimizing manual input (sync, rules, defaults over forms).
- Non-negotiable quality bar: reliability, data integrity, and security over feature count.
- Use Gemini where it materially helps (unknown merchants, receipts, insights); always keep deterministic fallbacks.

Execution Priorities
1) Reliability first
- Prevent silent failures; expose actionable recovery states; never lose user category/rule corrections.

2) Low-friction automation
- Prefer Plaid sync + merchant rules + curated `merchant_db` before asking the user to categorize.

3) Practical AI integration
- Gemini for receipts and insights; server-side categorization on sync when possible; client fallbacks when AI fails.

4) Incremental delivery
- Edge Functions verified on Plaid Sandbox before relying on iOS flows; compile on CI before TestFlight.

Core Product Rules
1) Plaid and secrets
- iOS → Supabase Edge Function → Plaid only. Never call Plaid API from iOS.
- Never store Plaid `access_token`, `client_secret`, or full account numbers on device (mask = last 4 only).
- Plaid `access_token` lives in Supabase Vault only.
- Plaid production readiness: follow `docs/PLAID_PRODUCTION_CHECKLIST.md` (webhooks, update mode, OAuth redirect URI, item removal).
- Edge Functions: 7 total — link, update-link, exchange, accounts, sync, remove-item, webhook (webhook uses Plaid JWT verification, not Supabase JWT).

2) Auth and privacy
- Face ID / Touch ID required before any financial screen (cold open and return from background or app switcher).
- 6-digit app PIN required on first use; after 3 biometric failures (or user choice), PIN unlock. Verifier in Keychain (PBKDF2), never plaintext PIN.
- `NSAllowsArbitraryLoads` = false in Info.plist.

3) Categorization priority (locked)
- (1) `merchant_rules` (user) → (2) `merchant_db` (seed) → (3) Gemini JSON → (4) Plaid raw category.
- Category enum is fixed (17 categories in PROJECT_BRIEF); validate AI JSON before saving.

4) Tabs and information architecture
- Bottom tabs: Dashboard, Transactions, Budgets, Goals, Insights.
- Bank link via Edge Functions only (`PlaidLinkView`); settings (rules, notifications) from gear / Profile path.

5) Budgets and goals
- Budget math and alert thresholds must stay deterministic and tested (`BudgetMath`, `BudgetAlertEngine`).
- Debt payoff strategies (avalanche/snowball) are client-side models; do not send debt balances to third parties.

6) Stability expectations
- Sync and link errors must surface in UI (`TransactionStore.errorMessage`, Plaid link status).
- User-edited categories must persist and not be overwritten on sync unless still `Other`.

Engineering Standards (backend, tests, structure)
1) Test-driven backend development
- “Backend” = data/services under `BudgetTracker/Backend/` (AI, Auth, Cloud, Core, Finance, Plaid).
- For new pure logic (categorization, budget math, goals, debt, cash flow, subscriptions): add or update tests in `BudgetTrackerTests/Backend/<domain>/` **first**, then implement until tests pass.
- UI-only changes in `BudgetTracker/Views/` need tests when they embed pure logic; prefer extracting logic into `Backend/Finance/`.

2) File size (AI-readable)
- Target ≤ 400 lines per Swift file; hard stop at 500 unless splitting would harm clarity.
- When a file exceeds 400 lines, split by responsibility (models vs store vs persistence vs API client).
- Run `powershell -File scripts/check-file-line-limits.ps1` before large merges.

2b) Pre-ship test gate (required before TestFlight)

## Why -xcconfig is mandatory
pkg.xcconfig contains:
  CODE_SIGNING_ALLOWED = NO
  CODE_SIGNING_REQUIRED = NO
  CODE_SIGN_IDENTITY =
  PROVISIONING_PROFILE_SPECIFIER =

Without it, Swift packages (Supabase, LinkKit) inherit distribution signing flags and fail with “does not support provisioning profiles”.
Per-target signing is in `project.yml`:
  - App → `Budget Tracker Distribution` profile

## Critical Rules
- NEVER pass PROVISIONING_PROFILE_SPECIFIER on the xcodebuild command line
- NEVER remove `-xcconfig pkg.xcconfig` from Release archive
- ALWAYS unlock keychain before archive (Step 1) on local Mac
- ALWAYS run `xcodegen generate` after editing `project.yml` (Step 2)
- ALWAYS increment `CURRENT_PROJECT_VERSION` before each TestFlight upload
- Build number must be strictly increasing — check App Store Connect first
